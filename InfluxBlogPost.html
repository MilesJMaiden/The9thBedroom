Abstract

Influx is a maintenance-and-sabotage simulation built in Unity 2022.3.56f1 (URP) where dungeon-style levels are procedurally generated via frontier-based room placement and NavMesh linking. Autonomous Agents sustain two interdependent resource systems—power (Computers) and fuel (Bins)—while hostile Aliens opportunistically sabotage the most vulnerable component. Agents use a prioritized finite-state machine (FSM) for repair, refuel, and alien capture, but remain fully interruptible via direct player commands. Interactables rely on coroutines for gradual depletion and sabotage; the HUD aggregates system health in real time; and a unified pause manager cleanly suspends both simulation and UI updates. The architecture emphasizes modularity and extensibility via DRY and SOLID design principles.

Key Contributions

Procedural Level Generation

Frontier-based room placement, corridor stitching, NavMesh baking, and NavMeshLinks for seamless traversal.

Performance & Stability

Object pooling for static geometry and robust “retry on failure” generation when spawn validation fails.

Safe-Area Spawning

Inset spawn regions plus physics overlap checks to prevent collisions and wall proximity issues.

Emergent Multi-Agent Loop

Agents repair, refuel, chase, trap, and respond to player overrides—while Aliens sabotage and flee.

Room-Level Alerting

Per-room monitoring triggers alarms + alien spawns when all computers in a room fail; HashSets avoid duplicates.

Unified HUD & Pause

Aggregated system health with pause/game-over handling that freezes simulation and UI updates coherently.

Level Design Settings

The procedural generator is driven by a ScriptableObject, enabling rapid iteration without code edits. Room shapes, room sizes, corridor connectivity, and per-room spawn quotas are authored as data.

Data-driven room shapes and spawn quotas

// LevelDesignSettings (ScriptableObject)
[CreateAssetMenu(fileName="LevelDesignSettings", menuName="Level/Design Settings")]
public class LevelDesignSettings : ScriptableObject {
  [Header("Room Shape Distribution")]
  public RoomShape defaultRoomShape = RoomShape.Square;
  [Range(0f,1f)] public float probabilitySquare = 0.5f;
  public float probabilityLShaped = 0.3f;
  public float probabilityTShaped = 0.2f;

  [Header("Room Size (tiles)")]
  public Vector2 defaultRoomSize = new Vector2(5, 5);
  public Vector2 minRoomSize     = new Vector2(4, 4);
  public Vector2 maxRoomSize     = new Vector2(8, 8);

  [Header("Connectivity")]
  [Range(0f,1f)]
  public float connectionProbability = 0.7f;

  [Header("Object Spawn Settings")]
  public RoomSpawnSettings[] roomSpawnSettings;
}

Interactables

All clickable world objects inherit from a single Interactable base class that encapsulates hover highlighting, cursor changes, and click dispatch. This enforces a consistent interaction contract and removes duplicated input logic across object types.

Base interaction: hover + click dispatch

[RequireComponent(typeof(Collider))]
public abstract class Interactable : MonoBehaviour {
  protected GameObject _highlight;

  protected virtual void Awake() {
    var t = transform.Find("HighlightCylinder");
    if (t != null) {
      _highlight = t.gameObject;
      _highlight.SetActive(false);
    }
  }

  private void OnMouseEnter() { _highlight?.SetActive(true); }
  private void OnMouseExit()  { _highlight?.SetActive(false); }
  private void OnMouseDown()  => OnClicked();

  protected virtual void OnClicked() { }
}

Derived behaviors

  • Bin: fuel storage; depletes slowly normally, but drains rapidly during sabotage; accepts refined rock deposits via Agent commands.
  • Computer: power system; degrades randomly over time; repair pauses depletion; sabotage overrides and drains at a fixed rate.
  • Container: drop-off for captured Aliens; Agents can deposit carried Aliens at a defined spawn point.
  • RockBin: provides raw rock; begins the Rock → Table → Bin refueling pipeline.
  • Table: refines rock over time (coroutine) with UI progress feedback.

Agents & Aliens AI

Influx is driven by two interacting AI systems: a prioritized multi-state Agent FSM for repair/refuel/capture, and an Alien FSM that seeks sabotage targets, attacks, flees when threatened, and wanders when idle.

Agent FSM: priority-driven decision loop

// Example of priority ordering (conceptual)
1) Deselect shortcut
2) Fuel depletion tick
3) Player override (only if selected and Idle/Wander)
4) If carrying Alien: trap/drop at Container
5) If in Rock pipeline: finish Rock → Table → Bin
6) Chase Alien if within range
7) Global refuel if Bin is low
8) Repair weakest nearby Computer
9) Fallback auto behaviors
10) Idle/Wander
Agents executing the Rock → Table → Fuel pipeline (RockRefining.png).

Alien FSM: seek → attack → flee → wander

// Core structure (conceptual)
enum State { Seek, Attack, Flee, Wander }

Update():
  - Clear spent targets (if slider <= 0)
  - Acquire a new sabotage target (Computer or Bin with value > 0)
  - Dispatch state handler:
      Seek: move to target
      Attack: sabotage while in range
      Flee: run away if Agent is nearby
      Wander: patrol random NavMesh points
Capture loop: Agent intercepts an Alien and carries it to a Container (AlienCapture1.png).

Level Generation & Room Alerts

The level generator constructs a tiled dungeon layout, spawns interactables per room, and connects rooms via corridors. Navigation reliability is enforced via local NavMeshes, corridor NavMeshLinks, and a final bake.

Build pipeline overview

  1. Layout: frontier-based room placement and grid positioning.
  2. Room GameObjects: create containers for each room and place them in world space.
  3. Geometry: generate floors and perimeter walls/doors; pool geometry for performance.
  4. Object spawning: place Computers/Bins/Containers/Displays within safe areas; abort+retry if validation fails.
  5. Alerts & Agents: attach per-room alert controllers, connect rooms via NavMeshLinks, bake NavMeshes, then spawn Agents.

Safe-area spawning (collision + margin validation)

// Safe area inset (conceptual)
SafeArea ComputeSafeArea(float margin) {
  float totalOffset = margin + defaultSafeOffset + wallMargin;
  return new SafeArea(
    new Vector2(totalOffset, totalOffset),
    new Vector2(roomW - totalOffset, roomH - totalOffset)
  );
}

// Validation (conceptual)
- OverlapBox against prefab collider extents
- Keep distance from Walls (OverlapSphere)
- Enforce spacing between large objects / Computers / Containers

Room alerts: spawn Aliens when all Computers fail

Each room monitors its Computer sliders. If all hit zero, the room enters an alert state: a red light flashes, an alarm loops, and Aliens spawn at Containers (tracked by a HashSet to prevent duplicates). The alert ends when any Computer is repaired above zero.

HUD & Pause Management

The HUD aggregates system health into three values: average Computer health, average Bin fuel, and an overall status. This keeps the player’s attention on macro-level stability rather than micromanaging individual sliders.

Global HUD: Computers, Bins, Overall system status, plus pause/menu overlay (hud.png).

Aggregation logic (conceptual)

Design Principles & Challenges

Principles applied

  • SRP: isolate responsibilities (spawning, generation, UI, AI states) into dedicated classes.
  • Dependency inversion: systems depend on abstractions (e.g., pools/spawners) rather than concrete implementations.
  • DRY: shared operations like safe-area computation and validation are reusable utilities.
  • Coroutine encapsulation: time-based behaviors live in coroutines rather than bloating Update loops.

Key challenges solved

  • Procedural layout consistency: grid alignment + corridor stitching + NavMeshLinks without navigation dead-ends.
  • Safe-area spawn validation: balancing randomness with strict no-overlap and minimum wall margins.
  • Repeatable room alerts: spawning Aliens once-per-container per alert cycle, then resetting cleanly.
  • UI state coherence: multiple canvases coordinated by a single manager.

Future Work

Based on playtester feedback and the project roadmap, the next iteration focuses on reliability and richer feedback, then expands into adaptive difficulty and broader content.

  • Bug fixes & AI tuning
    • Fix alert looping-sound issues when multiple alerts reset simultaneously.
    • Ensure sabotage consistently depletes Computer progress as intended.
    • Improve Agent transitions and scale Alien behavior in higher difficulty modes.
    • Ensure Bin “fast depletion” only occurs while actively sabotaged.
    • Improve player-initiated refueling pipeline reliability.
  • Adaptive emergent difficulty
    • Scale Alien spawn/sabotage intensity based on player performance metrics.
  • Expanded content & interactables
    • Add new systems (hazards, terminals, emitters) with bespoke sabotage/repair loops.
    • Increase prop variety per room to improve visual identity across seeds.
  • Enhanced visual & audio feedback
    • URP upgrades + contextual VFX and clearer audio cues.
  • Multiplayer co-op
    • Multiple players controlling Agents concurrently with synchronized commands.
  • Data-driven hot-reload
    • Externalize configuration to JSON/CSV and support runtime hot-reloading for rapid tuning.

Conclusion

Influx demonstrates a modular Unity framework that combines procedural generation, coroutine-driven interactables, and prioritized AI state machines to produce an emergent “maintenance vs. sabotage” simulation. A data-driven generator enables rapid iteration, safe-area spawning improves robustness, room-level alerts create tension spikes, and the HUD + pause systems maintain UI clarity during chaos. The design supports extension: new interactables, richer content, adaptive difficulty, and multiplayer can be added with minimal friction while preserving the emergent behavior players found most compelling.
