Tetroid: A Heuristic Tetris AI using Beam Search Weighted features + discounted lookahead planning (d=3, B=3, γ=0.9)

This post summarizes Tetroid, an autonomous Tetris-playing AI that evaluates board states using a weighted heuristic feature vector and selects moves with a recursive beam search. The goal is to balance short-term line clears with long-term stability by exploring future placements and maximizing a discounted cumulative score.

Abstract

Tetroid is a heuristic-driven AI agent designed to play :contentReference[oaicite:0]{index=0} autonomously. It evaluates candidate placements using a weighted feature function (e.g., lines cleared, aggregate height, holes, bumpiness, wells, center deviation, landing height, locked spaces) and selects moves via a recursive beam search that plans ahead across upcoming pieces. By maximizing a discounted cumulative score, Tetroid avoids greedy line clears that create long-term instability, while still prioritizing survival-critical clears.

Why Tetris is Hard

Tetris is challenging for AI because decisions have delayed consequences: a placement that clears a line now can introduce holes or awkward geometry that becomes fatal several pieces later. The branching factor is large (rotations × columns × drop outcomes), and the game is effectively unbounded in time, meaning “good” play is survival-oriented rather than reaching a terminal win state.

Historically, strong approaches often combine a compact, interpretable feature set with search: a good evaluation function can outperform more complex strategies in this domain when paired with lookahead. Tetroid follows that tradition—treating Tetris as “evaluate-and-plan” rather than learning a policy end-to-end.

Heuristic Features and Weights

Tetroid scores a post-placement board state using a linear evaluation function. Positive weight rewards progress (line clears), while negative weights penalize structural risk (height, holes, uneven terrain, trapping). 

E(state) =
  w_lines      * lines_cleared
+ w_aggHeight  * aggregate_height
+ w_holes      * holes
+ w_bumpiness  * bumpiness
+ w_wells      * wells
+ w_centerDev  * center_deviation
+ w_landHeight * landing_height
+ w_locked     * locked_spaces
Feature Weight Intent
Complete lines +0.760666 Reward clears (survival + score proxy).
Aggregate height -0.510066 Penalize tall stacks (reduces top-out risk).
Holes -0.356630 Strongly penalize inaccessible empty cells under blocks.
Bumpiness -0.184483 Prefer flatter surfaces (more placements remain legal).
Wells -0.500000 Discourage deep single-column pits that become traps.
Center deviation -0.100000 Promote balance (avoid lopsided stacks).
Landing height -0.050000 Prefer lower lock-in to preserve headroom.
Locked spaces -0.200000 Penalize trapped blocks above holes (burial risk).

Design rationale

The first four weights match the classic “El-Tetris” style vector (clear lines; avoid height/holes/roughness). Wells and landing height align with Dellacherie-style feature ideas. Center deviation and locked spaces act as practical stabilizers: they bias away from brittle geometries that a shallow search horizon can’t always repair in time.

Recursive Beam Search Strategy

Exhaustively searching all move sequences is infeasible, so Tetroid uses beam search to retain only the top B states per depth. This provides lookahead planning while keeping computation bounded.

  1. Enumerate all legal placements for the current piece.
  2. Expand successors up to depth d, but keep only the best B states at each layer.
  3. Score sequences using a discounted sum with γ = 0.9.
  4. Execute the first move of the best sequence; repeat every piece.
S = E(s1) + γ E(s2) + γ² E(s3) + ... + γ^(d-1) E(sd)

Chosen parameters

  • d = 3 (lookahead depth)
  • B = 3 (beam width)
  • γ = 0.9 (discount factor)

This configuration gives enough foresight to avoid many “greedy traps” (like quick clears that create holes), while staying lightweight enough to run interactively.

Results and Performance

Tetroid was evaluated over 30 independent games. The distribution of final scores shows a wide spread (expected in Tetris due to piece sequence variance and horizon effects), with a median score of 114,701 and a mean score of 188,893.5.

Histogram of final scores over 30 runs (HistogramResults30.png).

Qualitative behavior

  • Surface shaping: aims for flatter stacks; fixes spikes early (bumpiness penalty).
  • Hole aversion: strongly avoids creating holes; repairs quickly when unavoidable.
  • Non-greedy clears: sometimes delays a clear if the lookahead sees a better stabilizing sequence.
  • Occasional “clever” placements: overhang-like structures can appear when the search predicts resolution.

Discussion: What Worked Well

  • Line clearing as a survival primitive: a strong positive line-clear reward keeps board height manageable and reduces risk.
  • Holes as the “main enemy”: penalizing holes produces a noticeable improvement in stability; most losses occur after hole accumulation or burial.
  • Beam search foresight: even a depth of 3 often prevents immediate gratification moves that set up dead boards a few pieces later.
  • Interpretable control: feature weights make behavior explainable and debuggable—useful for iterative refinement.

Failure Modes and Limitations

1) Illegal/overlapping placements (simulation validity)

Occasionally the AI attempts moves that result in overlapping pieces or invalid placements (often near walls/rotations), indicating a bug or edge case in the move generator / collision validation rather than the heuristic itself. This can cause avoidable losses or corrupted evaluations.

2) Shallow horizon (search depth)

With d=3, Tetroid can be surprised by longer-term dependencies (e.g., building a well expecting an I-piece soon, but the required piece arrives later than the horizon). The discount factor can also reduce the perceived value of “temporary ugliness” that pays off after more than 3 moves.

3) Weight balance edge cases

Strong penalties (height/holes) can sometimes over-correct: the AI may avoid a short-lived hole that would be trivially repaired next piece, particularly if the repair sits just outside the horizon or is discounted heavily.

4) No adaptation / learning

The agent’s weights are static. Without optimization or learning, the AI repeats the same failure patterns rather than tuning away from them over time.

Improvements Roadmap

Area Proposed improvement Expected effect
Move validity Harden collision + rotation logic; unit tests across wall-kicks / edge placements. Removes illegal moves and improves evaluation reliability.
Search Increase d/B where feasible; consider iterative deepening or selective expansion. Better long-horizon planning; fewer “surprise” deaths.
Features Add transitions (row/column), more nuanced well metrics, pattern detectors (e.g., safe Tetris setup). Improves discrimination between superficially similar boards.
Weight tuning Offline optimization (GA / cross-entropy / RL) to refine weights for your exact ruleset. Higher scores + stronger stability without manual guesswork.
Modern mechanics Add hold-piece branching and preview utilization (if ruleset supports it). Much stronger tactical flexibility; fewer forced bad placements.

Pragmatic next step

Fixing overlap/invalid move generation yields the biggest immediate win because it upgrades the integrity of every downstream decision: the heuristic and search are only as good as the simulator.
