A Modular Cellular Automata Framework in Unreal Engine 5.5

This post documents a custom, modular implementation of Conway’s Game of Life built in Unreal Engine 5.5 using C++ for simulation and Blueprint-exposed parameters for iteration, visuals, and pattern orchestration. The result is a flexible framework for classic emergent behaviours—still lifes, oscillators, and spaceships—augmented with smooth opacity fading and optional 3D mesh animation.

Abstract

This project presents a custom and modular implementation of Conway’s Game of Life inside Unreal Engine 5.5. The framework combines C++ classes for efficient simulation with Blueprint-exposed parameters and real-time visual effects. A central ACellularAutomataManager maintains grid state, performs neighbour-rule updates, spawns cell actors dynamically, and applies a library of well-known patterns (still lifes, oscillators, spaceships). Visual enhancements—opacity fading and optional mesh transforms— provide smooth iteration and a clear read on emergent behaviour. The outcome is a designer-friendly foundation that reproduces Life’s classic dynamics while staying open to future expansions (alternate rule sets, scenarios, UI tooling, and 3D variants).

Introduction

Cellular automata are discrete computational systems where simple local rules can yield complex emergent behaviour at scale. Among these, the Game of Life—introduced by :contentReference[oaicite:1]{index=1} in 1970—remains a canonical example, known for stable “still life” structures, periodic oscillators, and moving “spaceships”.

Life rules (neighbour-based constraints)

  • A live cell with fewer than two neighbours dies (under-population).
  • A live cell with more than three neighbours dies (overpopulation).
  • A live cell with two or three neighbours survives.
  • A dead cell with exactly three neighbours becomes alive (reproduction).

Aims & motivation

  • Efficient + modular: C++ simulation core with designer-friendly Blueprint controls.
  • Extensible patterns: add/modify patterns without changing the simulation engine.
  • Readable visuals: opacity fading, subtle animations, and clear pattern identification.

Program Overview

The prototype runs as a self-contained Unreal level. On Play, the manager initializes the grid, optionally spawns multiple seeded patterns, and advances the simulation at a configurable timestep. Designers can tune grid size, update interval, fade duration, and pattern spawn settings from Blueprint.

ACellularAutomataManager

Owns grid state, rules, timing, and cell actor lifecycle. Exposes initialization, tick-driven update, and pattern application to Blueprints.

Pattern actors

Pattern classes seed grid indices using offsets (Block, Beehive, Loaf, Blinker, Toad, Beacon, Glider, LWSS, MWSS).

Blueprint hooks

Pattern selection, counts, and simulation parameters are editable in Blueprint for rapid iteration without recompiling.

Dynamic visuals

Cells are rendered via mesh + material. Opacity, colour, and optional transforms update smoothly to communicate births/deaths and motion.

Class Overview

All patterns available in the application (Images/CAPatterns.png).

The project structure maps Blueprint assets to C++ types: Blueprints reside in All/Content/MyBlueprints, derived from matching C++ classes under C++Classes/PGE2/Public. Meshes live in All/Content/MyBasicMeshes, materials in All/Content/MyMaterials, and the main level in All/Content/Levels.

Core types

  1. ACellularAutomataManager — maintains a 2D grid (TArray<int32>), updates cell states each step, and spawns/destroys mesh actors for live cells. Exposes functions like InitializeGrid(), UpdateSimulation(), and ApplyPattern().
  2. ACellPatternBase — abstract base for patterns. Subclasses provide offsets for still lifes, oscillators, and spaceships. New patterns extend the library without modifying the manager (OCP-friendly).

The architectural split is intentional: the manager owns the simulation loop and grid, while patterns are small, declarative “shape” definitions. This makes the system easy to extend (add a new pattern class) without rewriting simulation code.

Manager implementation (pattern spawning + seeding)

void ACellularAutomataManager::BeginPlay()
{
    Super::BeginPlay();
    InitializeGrid();

    for(const FPatternSpawnInfo& Info : PatternSpawnInfos)
    {
        for(int32 i = 0; i < Info.InstanceCount; i++)
        {
            int32 RandX = FMath::RandRange(0, GridWidth - 1);
            int32 RandY = FMath::RandRange(0, GridHeight - 1);

            FVector PatternOrigin = GetActorLocation() +
                FVector((RandX + 0.5f)*100.f, (RandY + 0.5f)*100.f, 0.f);

            ApplyPattern(Info.PatternClass, PatternOrigin);
        }
    }
}

void ACellularAutomataManager::ApplyPattern(TSubclassOf<ACellPatternBase> PatternClass, FVector Origin)
{
    if(!PatternClass) return;

    ACellPatternBase* Pattern = GetWorld()->SpawnActor<ACellPatternBase>(
        PatternClass, Origin, FRotator::ZeroRotator);

    if(Pattern)
    {
        Pattern->ApplyPattern(this);
        ActivePatternActors.Add(Pattern);
    }
}

Implementation Details

Blueprint math expression node for interpolating opacity over time (Images/CAFadingPatterns.png).

Math Expression nodes

  1. Cell fade calculation: an ease-out opacity curve (quadratic) to avoid harsh popping: opacity = 1 - (activationTime / fadeTime)^2.
  2. Pattern rotation: a time-normalised rotation for spaceship visuals: angle = 360 * (timeElapsed / cycleDuration).

DRY + SOLID in practice

  • DRY: shared pattern-application logic lives in ACellPatternBase; patterns mostly define offsets.
  • SRP: manager = grid/rules/timing; patterns = shapes; materials = visuals.
  • OCP: add new patterns by subclassing, without editing the manager.
  • LSP: any pattern subclass can be spawned/used through the base type.
  • ISP: lean, focused interfaces for manager vs patterns.
  • DIP: manager depends on the abstract base (not concrete patterns).

Challenges & solutions

Challenge: large grid performance

Early prototypes dropped frames beyond ~100×100 due to frequent spawn/destroy churn.

Solution: object pooling + partial updates. Reuse actors and update only changed cells to reduce allocation and GC overhead.

Challenge: sync destruction with fade

Immediate destruction cancels the fade-out, causing abrupt visuals.

Solution: delay destruction until the fade completes (timeline + callback), or use SetLifeSpan as a controlled teardown.

Future Work

Scenario system

Replace random placement with curated scenario definitions (pattern type, counts, and explicit coordinates) to support demos, educational setups, and reproducible experiments (e.g., glider collisions).

Interactive UI (UMG)

  • Runtime controls for grid size, timestep interval, fade/rotation toggles.
  • Pattern selection + spawn controls without returning to the editor.
  • Start/pause/reset controls with clear state feedback.

Alternate rules + 3D

Extend beyond Conway’s rule set (HighLife, Day & Night), and explore volumetric/3D variants with new rendering and camera constraints.

Performance roadmap

For very large grids: parallel updates (task graph) and/or GPU compute (compute shaders), plus instanced mesh rendering to cut draw calls.

Conclusion

Snapshot of emergent behaviour patterns (Images/EmergentCAPatterns.png).

This work demonstrates a modular, extensible approach to Conway’s Game of Life in Unreal Engine 5.5. The manager governs grid updates and lifecycle, while pattern subclasses declaratively seed structures (gliders, oscillators, still lifes). Blueprint exposure enables real-time tuning, and dynamic material effects (fade/colour/optional transforms) improve readability and aesthetic continuity. Even as a 2D simulation, the framework is well-positioned for richer expansions—3D automata, alternate rules, curated scenarios, and interactive UI—showcasing how classic algorithmic systems benefit from modern engine tooling.
